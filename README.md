# 晚报行情错位问题定位报告

日期：2026-03-12  
作者：小雷

---

## 一、问题概述

用户反馈：`2026-03-11 晚报` 中个股价格与涨跌幅看起来不对，怀疑报告生成链路存在行情错位或脏数据问题。

本次定位目标：
1. 确认错在展示层、API 层还是生成层
2. 确认价格对应的实际交易日
3. 确认是否存在更底层的数据源异常
4. 在**不改代码**的前提下，给出链路级结论

---

## 二、核心结论

### 结论 1：问题不在展示层，也不在 `api.py latest evening`
当前用户拿到晚报的链路是：

1. `skills/public/trading-api/scripts/api.py latest evening`
2. 调 `/api/v1/reports/` 找最新晚报
3. 调 `/api/v1/reports/{id}` 取报告详情
4. 直接输出数据库中已经生成好的 `content_markdown`

因此，个股价格不是展示时实时算出来的，而是在报告生成时就已经写入正文。

### 结论 2：`2026-03-11 晚报` 中的个股价格实际对应的是 **2026-03-10 美股收盘**
通过直接查询本地 trading API quote 数据验证：

- NVDA `184.77 (+1.16%)` 对应 `2026-03-10 close` vs `2026-03-09 close`
- META `654.07 (+1.03%)` 对应 `2026-03-10 close` vs `2026-03-09 close`
- AAPL `260.83 (+0.37%)` 对应 `2026-03-10 close` vs `2026-03-09 close`
- TSLA `399.24 (+0.14%)` 对应 `2026-03-10 close` vs `2026-03-09 close`
- MSFT `405.76 (-0.89%)` 对应 `2026-03-10 close` vs `2026-03-09 close`
- ORCL `149.40 (-1.43%)` 对应 `2026-03-10 close` vs `2026-03-09 close`

也就是说：

> 表格里的数字本身并没有算错，但它们对应的是 **03-10 的 official daily close**，不是读者直觉理解的“03-11 晚报当天盘面”。

### 结论 3：问题发生在**报告生成阶段**，不是读取阶段
直接拉取 `/api/v1/reports/297`（`2026-03-11 晚报`）原始内容，`content_markdown` 中已经内嵌了上述价格。

这证明：
- trading-api 返回的是数据库已有内容
- 报告价格错位在报告生成时就已经发生
- 不存在前端/展示层二次拼接出错

### 结论 4：根因是“北京时间报告日期”和“美股行情交易日”被混用了
报告元数据：
- report id: `297`
- report_date: `2026-03-11`
- report_type: `evening`
- created_at: `2026-03-11T13:17:03Z`
- 对应北京时间：`2026-03-11 21:17`

而这个时间点：
- 北京时间 21:17 时，美股 `2026-03-11` 正式交易日的日线尚未形成
- 生成器能拿到的最新 official daily bar 只能是 `2026-03-10 close`

所以服务端实际上做的是：

> 按北京时间生成“03-11 晚报”，但价格快照取的是美股上一交易日收盘（03-10 close）

这会在语义上造成错觉：
- `report_date` 看起来是 03-11
- 价格却还是 03-10 close
- 如果正文又在讲 03-11 当晚的新事件，用户会天然觉得“价格错了”

### 证据 A：report 297 的正文里已写死这些价格
从 `/api/v1/reports/297` 的 `content_markdown` 可直接读到：

- NVDA `$184.77 (+1.16%)`
- META `$654.07 (+1.03%)`
- AAPL `$260.83 (+0.37%)`
- TSLA `$399.24 (+0.14%)`
- MSFT `$405.76 (-0.89%)`
- ORCL `$149.4 (-1.43%)`

其中 ORCL 还被报告正文显式标注：

> `数据时间 2026-03-10 00:00 | 🔴 数据较旧（37小时前）`

说明生成器自己都知道 ORCL 数据旧，但仍将其写入正式报告。

### 证据 B：本地 quote API 验证价格对应 03-10 close
直接通过本地 trading API 查询 DAILY bar：

- NVDA
  - 2026-03-11 close = `186.03`
  - 2026-03-10 close = `184.77`
- META
  - 2026-03-11 close = `654.86`
  - 2026-03-10 close = `654.07`
- MSFT
  - 2026-03-11 close = `404.88`
  - 2026-03-10 close = `405.76`
- ORCL
  - 2026-03-11 close = `163.12`
  - 2026-03-10 close = `149.40`

这与 report 297 的文本完全对齐。

### 证据 C：报告创建时间早于当晚正式收盘数据可用时间
`created_at = 2026-03-11T13:17:03Z` = 北京时间 `21:17`

在该时间点，美股 03-11 的日线 close 不可能已经完整落库，因此如果生成器用的是 daily close，就只能回退到 03-10。

## 四、额外发现

### 1. 指数数据源本身也有严重过期问题
report 297 正文中明确提到：

- 核心 ETF（SPY/QQQ/IWM）MCP 数据获取失败
- IBKR 指数数据日期为 `2026-02-13`
- 被报告作者/生成器标注为“23天前，严重过时”

这说明除“日期语义错位”外，还存在**指数源可用性异常**。

### 2. AMD 行情源存在独立滞后
本地 quote API 查询显示 AMD 仅返回：
- `2026-03-03 close = 198.62`

说明 AMD 不是单纯晚一天，而是 ticker 级别的行情同步异常。

### 3. stock_analyses 结构并未保存价格元字段
在 `/api/v1/reports/297` 返回的数据中，`stock_analyses` 虽然有 ticker（META / MSFT / ORCL / NVDA），但并没有填充 `current_price`、`price_change_pct` 等数值字段。

这意味着：
- 价格主要体现在 `content_markdown` 自由文本里
- 价格快照日期没有独立字段暴露给客户端
- 客户端也无法轻松识别“价格对应的交易日”

## 五、责任边界判断

### 可以排除的层
- Telegram 展示层
- OpenClaw 消息发送层
- `api.py latest evening` 客户端入口
- `/api/v1/reports/{id}` 的读取逻辑

### 高概率责任层
- **服务端报告生成器 / prompt 组装器 / 数据快照选择逻辑**

更具体地说，问题像是以下逻辑设计导致：

1. 用北京时间决定 `report_date = 2026-03-11`
2. 用“当前可得最新日线”填个股价格
3. 这个 latest daily bar 实际是 `2026-03-10 close`
4. 但没有把 `price_snapshot_date` 作为独立元数据保存/展示
5. 最终形成“03-11 晚报 + 03-10 收盘价”的混合产物

---

## 六、问题本质

这不是一个简单的“算错价格”问题，而是一个**报告日期语义与行情快照日期语义没有分离**的问题。

用更准确的话说：

> 服务端生成的是“北京时间 03-11 晚上生成的一份报告”，但报告里的个股价格实际是“美股上一交易日 official close”。

当正文又混入 03-11 当晚的新叙事（战争、黄仁勋文章、财报、风格切换）时，阅读体验就会像“价格写错了”。

## 七、后续定位建议（本报告不改代码）

如果继续追查，建议下一步查这两项：

### 路径 A：反查 MCP 标签来源
report 297 中价格引用为：
- `[MCP_006]` NVDA
- `[MCP_009]` MSFT
- `[MCP_011]` ORCL
- `[MCP_014]` META

下一步可追：
- 这些 `MCP_00x` 在服务端生成时到底对应哪类 quote source
- 是直接取 `/api/v1/quotes/{symbol}` 还是别的中间层缓存
- 是否带有 source timestamp

### 路径 B：查生成器对“晚报”的定时规则
重点看：
- 报告生成时间为何是北京时间 21:17
- 是否设计上就默认“晚报 = 北京时间晚上生成”
- 是否没有区分“盘前晚报”和“收盘后晚报”

---

## 八、最终结论（一句话）

**问题已定位到服务端报告生成链路：`2026-03-11 晚报` 使用了 `2026-03-10` 的美股日线收盘快照，根因是“北京时间报告日期”和“美股行情交易日”混用，且价格快照日期没有被结构化保存与显式标注。**
